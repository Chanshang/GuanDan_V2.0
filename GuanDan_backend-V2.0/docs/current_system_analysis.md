# 当前系统分析

## 1. 现有业务流程

根据当前 `run.py` 的逻辑，系统主要流程如下：

1. 上传报名 Excel
2. 解析参赛办公室、队长、4 个子队、等级和队员信息
3. 初始化数据库中的队伍信息、队伍等级信息、3 轮初始积分信息
4. 随机生成 3 轮对阵
5. 设置当前轮次 `TURN`
6. 用户按桌号查询自己本轮比赛
7. 录入双方最终等级结果
8. 系统计算大小分并更新排行榜
9. 前端轮询接口获取对阵、计时、排名、办公室排名

## 2. 当前代码中的主要模块

虽然现在都写在 `run.py` 中，但从职责上已经可以拆成几类：

### A. 基础状态控制

- `TURNsinfo`
- `set_turn`
- `start_timer`
- `stop_timer`
- `timesinfo`

说明：

- 当前使用全局变量 `TURN` 和 `timer_start`
- 这种方式适合单机原型，不适合多进程部署

### B. 数据展示接口

- `matchesinfo`
- `scoresinfo`
- `sumteaminfo`
- `officescore`

说明：

- 这些接口已经具有 API 雏形
- 很适合未来保留为 REST 接口

### C. 数据初始化

- `index`
- `clear_tables`

说明：

- Excel 导入、字段清洗、批量写库都在这里
- 这是后续最应该先独立成 service 的部分

### D. 对阵生成

- `generate_matches`

说明：

- 这是当前系统最核心的业务规则之一
- 规则包括：
  - 3 轮比赛
  - 同轮每队只出现一次
  - 不同办公室优先
  - 避免重复对阵
  - 尽量覆盖不同等级匹配

### E. 成绩录入

- `select_table`
- `modify_select_table`
- `input_scores`
- `save_log`

说明：

- 已经形成了完整的录分闭环
- 但录分规则、参数校验、数据库更新还没有抽离

## 3. 当前数据模型理解

从代码推测，当前数据表职责大致如下：

- `team_info`
  - 报名原始信息
  - 一个办公室对应若干支子队

- `team_level`
  - 每支子队的等级标签

- `mini_team_info`
  - 每支子队在每一轮的得分记录
  - 当前是最核心的统计表

- `office_info`
  - 办公室维度的轮次得分
  - 目前代码里初始化了，但实际聚合多从 `mini_team_info` 直接计算

- `fight_info`
  - 每轮桌号与对阵关系

## 4. 当前主要问题

### 4.1 架构问题

- 所有逻辑都集中在 `run.py`
- 路由函数直接操作 SQL
- 页面模板与 API 接口混在一起
- 全局变量驱动状态，不利于并发和部署

### 4.2 测试问题

- 缺少独立的业务函数
- 对阵生成规则难以单元测试
- 录分逻辑只能靠页面手工点击验证
- 缺少固定测试数据集

### 4.3 可维护性问题

- 导入逻辑、匹配逻辑、积分逻辑都较长
- 规则变化时修改成本高
- 未来加入前端后，接口边界不够清晰

### 4.4 工程问题

- 项目中当前未看到完整的 `app/` 目录，但 `run.py` 引用了 `app.config`
- 模板内容存在编码显示异常现象，后续建议统一 UTF-8

## 5. Flask 还是 FastAPI

## 结论

短期内：

- 如果你现在的主要目标是继续打磨业务规则，Flask 可以继续用

中期内：

- 如果你准备做前后端分离、接口文档、自动化测试、规范请求响应模型，FastAPI 更合适

## 为什么 FastAPI 更适合后续阶段

- 更适合纯后端 API 项目
- 原生支持请求参数校验
- 自带 OpenAPI / Swagger 文档
- 更方便前后端联调
- 对接口测试和数据模型定义更友好

## 为什么现在不用急着立刻迁移

- 当前最大问题不是框架性能，而是代码职责混杂
- 如果直接迁移，很多耦合逻辑会原样搬过去
- 更合理的做法是先把业务逻辑抽成独立函数和服务层，再平滑迁移

## 6. 没有前端时如何测试功能

即使没有正式前端，你也完全可以测试后端功能。

建议按 3 层测试：

### A. 业务函数测试

测试纯 Python 函数，不依赖页面和数据库，例如：

- 对阵生成规则是否满足约束
- 比分输入后大小分计算是否正确
- 平局时最后一局胜者判定是否正确

### B. 接口测试

直接调用 HTTP 接口，不需要页面：

- `curl`
- Postman
- Apifox
- Flask 的测试客户端
- FastAPI 的 `TestClient`

### C. 数据层测试

准备一套最小测试数据：

- 4 个办公室
- 每个办公室 4 支队伍
- 固定等级分布

然后验证：

- 能否成功导入
- 能否生成 3 轮合法对阵
- 能否录分
- 排名是否正确

## 7. 当前最值得优先拆分的业务

建议优先拆以下 4 块：

1. Excel 导入服务
2. 对阵生成服务
3. 成绩录入与积分计算服务
4. 排名统计查询服务
