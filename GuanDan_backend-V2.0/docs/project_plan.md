# 项目重构与迭代计划

## 目标

在不打断现有业务的前提下，把当前原型项目逐步整理成“可维护、可测试、可扩展”的后端项目。

## 推荐迭代顺序

## 第一阶段：先稳住业务

目标：

- 不改页面交互
- 不换框架
- 先做可维护化

任务：

1. 新建 `app/` 目录
2. 抽离数据库连接配置
3. 抽离 Excel 导入逻辑
4. 抽离对阵生成逻辑
5. 抽离录分和积分计算逻辑
6. 抽离排名统计逻辑
7. 给关键服务补单元测试

完成标准：

- `run.py` 只保留路由和少量参数整理
- 核心规则可脱离页面独立测试

## 第二阶段：接口化

目标：

- 让系统既能服务模板页面，也能服务未来前端

任务：

1. 明确 API 输入输出格式
2. 区分页面路由和数据接口路由
3. 统一返回 JSON 结构
4. 统一错误码和错误信息

完成标准：

- 核心功能都可以直接通过 API 完成
- 前端不依赖模板也能联调

## 第三阶段：决定是否迁移 FastAPI

如果到这一步你准备：

- 独立前端项目
- 联调文档
- 自动接口文档
- 更强参数校验

就可以迁移到 FastAPI。

迁移时建议：

1. 保留 `services/` 不动
2. 仅替换最外层路由框架
3. 逐个把 Flask 路由迁成 FastAPI 路由
4. 用 Pydantic 补齐请求响应模型

这样迁移成本最低。

## 你现在最需要的测试方案

## 1. 单元测试

优先测试：

- 对阵生成是否满足约束
- 平局时 winner 逻辑是否正确
- 大分小分更新是否正确
- 排名排序是否正确

## 2. 接口测试

建议覆盖：

- 上传名单
- 生成对阵
- 设置轮次
- 查询某轮对阵
- 录入比分
- 查询当前排名

## 3. 回归测试数据

建议固定一份最小样本数据：

- 8 或 16 支队伍
- 覆盖不同办公室
- 覆盖不同等级
- 覆盖平局、超时、正常胜负

以后每次改规则都用这套数据回归。

## 推荐的项目管理方式

建议把需求拆成 4 条主线，而不是按页面拆：

### 主线 1：报名导入

- Excel 结构定义
- 字段清洗
- 入库校验

### 主线 2：对阵编排

- 编排约束
- 编排失败重试
- 可解释性输出

### 主线 3：录分结算

- 输入校验
- 大小分计算
- 日志记录

### 主线 4：统计展示

- 当前轮排名
- 总排名
- 办公室排名
- 桌号查询

## 建议的近期待办

1. 把 `run.py` 中对阵生成逻辑抽到独立 service
2. 把比分计算逻辑抽到独立 service
3. 给这两个 service 先补测试
4. 再考虑是否开始拆模板和 API

## 我对这个项目的判断

这个项目已经不是“从零开始”，而是“一个功能闭环已经跑起来的比赛管理原型”。最适合的策略不是推倒重来，而是：

- 先梳理规则
- 再拆模块
- 再补测试
- 最后决定框架迁移

这样风险最小，也最符合你现在这个阶段的投入产出比。
